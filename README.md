# 🧠 NPU Domain RAG System

A production-quality **Retrieval-Augmented Generation (RAG)** system built on a specialized dataset about the **History and Architecture of Neural Processing Units (NPUs)**. Powered by Groq's ultra-fast LLaMA 3.3 70B, FAISS vector search, and a custom multi-metric evaluation framework.

## 🎯 Chosen Domain & Dataset

**Domain:** History and Architecture of Neural Processing Units (NPUs)

This domain was chosen for three reasons:
1. **Specificity** — NPU architecture is narrow enough that a general LLM without retrieval would produce vague or outdated answers, making RAG genuinely valuable.
2. **Technical depth** — The documents contain precise numerical facts (TOPS figures, chip generations, year of release) that allow meaningful quantitative evaluation of factual correctness.
3. **Recency** — NPU developments through 2024 are at the frontier of LLM knowledge, stress-testing retrieval over pure parametric memory.

**Dataset:** 7 hand-crafted documents (~500–800 words each) covering:
- Origins and early history of NPUs
- Core systolic array architecture
- Data center NPUs (Google TPU, AWS Inferentia/Trainium, Cerebras WSE)
- Mobile & edge NPUs (Apple Neural Engine, Qualcomm Hexagon, MediaTek APU)
- NPU programming models (XLA, TVM, MLIR, ONNX)
- Benchmarking standards (MLPerf, Roofline model)
- Future NPU technologies (compute-in-memory, photonic computing, neuromorphic chips)

**QA Pairs:** 15 questions with detailed expected answers, mapped to their source documents for retrieval accuracy evaluation.

## 🔧 RAG Pipeline Design

### 1. Document Ingestion (`src/ingest.py`)

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Chunk size | 400 characters | Short enough to retrieve specific facts; long enough for context |
| Chunk overlap | 80 characters | 20% overlap prevents facts from being split across chunk boundaries |
| Embedding model | `all-MiniLM-L6-v2` | 384-dim, fast, high-quality semantic embeddings; runs locally, no API cost |
| Index type | FAISS `IndexFlatIP` | Exact inner-product search (≡ cosine similarity on unit-norm vectors); no approximation error on a small dataset |

**Pipeline:**
```
.txt files → load_documents() → chunk_document() → embed_chunks() → build_faiss_index() → save to disk
```

### 2. Retrieval (`src/retriever.py`)

The `Retriever` class encodes the query using the same `all-MiniLM-L6-v2` model, performs an exact FAISS search, and returns the top-k chunks with cosine similarity scores.

- **top_k = 5** by default — enough context for nuanced answers without overwhelming the prompt
- **`retrieve_with_dedup()`** offers source-level deduplication for broader coverage

### 3. Generation (`src/rag_pipeline.py`)

- **LLM:** Groq `llama-3.3-70b-versatile` — state-of-the-art 70B parameter model served at ~300 tokens/second via Groq's LPU inference hardware
- **Temperature:** 0.2 — low randomness for factual grounding
- **System prompt:** Explicitly instructs the model to answer only from context, cite sources, and flag insufficient context rather than hallucinate
- **Max tokens:** 512 — sufficient for detailed technical answers

**Prompt structure:**
```
SYSTEM: [Expert persona + grounding instructions]
USER:   CONTEXT: [numbered chunks with source labels and scores]
        QUESTION: [user query]
```

## 📊 Evaluation Framework

The evaluation framework (`src/evaluator.py`) runs all 15 QA pairs through the RAG pipeline and computes five metrics per question, producing a CSV report and console summary.

### Quantitative Metrics

#### 1. Semantic Similarity Score (weight: 30%)
Cosine similarity between the embedding of the generated answer and the expected answer, both encoded with `all-MiniLM-L6-v2`.

```
sem_sim = cosine_similarity(embed(generated), embed(expected))
Range: [0, 1] — higher is better
```

This is the primary metric because it captures meaning-level agreement rather than surface-level word overlap.

#### 2. ROUGE-1 & ROUGE-L F1 (weight: 25% combined)
Standard summarization metrics measuring unigram overlap (ROUGE-1) and longest common subsequence (ROUGE-L) between generated and expected answers.

```
ROUGE-1 F1 = 2 * (precision * recall) / (precision + recall)
             where precision/recall are over unigrams
```

ROUGE is well-understood and interpretable, though it penalizes valid paraphrasing.

#### 3. Keyword Hit Rate (weight: 20%)
Fraction of 15 domain-specific keywords (NPU, systolic, TOPS, inference, quantization, etc.) present in the generated answer.

```
keyword_hit_rate = |{kw : kw in generated}| / |DOMAIN_KEYWORDS|
```

This measures whether the answer uses appropriate technical vocabulary, catching cases where the model gives correct but vague non-technical answers.

#### 4. Retrieval Relevance Score (weight: 15%)
Mean cosine similarity score of the top-k retrieved chunks for the query.

```
retrieval_relevance = mean([chunk.score for chunk in retrieved_chunks])
```

Measures whether the retriever is finding genuinely relevant context, independent of LLM quality.

#### 5. Composite Score
Weighted combination of all quantitative metrics:

```
composite = 0.30 × sem_sim + 0.25 × rouge1_f1 + 0.20 × kw_rate
           + 0.15 × ret_rel + 0.10 × source_hit
```

### Retrieval Performance Metric

**Source Hit Rate:** Binary check — did the retriever return at least one chunk from the ground-truth source document for this question?

```
source_hit = expected_source in {chunk.source for chunk in retrieved_chunks}
```

Aggregated as a percentage across all 15 questions.

### Qualitative Assessment Rubric

Running `python main.py evaluate --qualitative` enables an interactive human-scoring mode. After each question, the evaluator displays:

1. The generated answer
2. The expected answer side-by-side

Then prompts for scores on four criteria (1–5 scale):

| Criterion | Description |
|-----------|-------------|
| **Coherence** | Grammatically correct and logically structured |
| **Completeness** | Covers all key aspects of the question |
| **Factual Correctness** | Facts match the reference and source documents |
| **Conciseness** | Appropriately tight without losing substance |

A `qualitative_avg` column is added to the results CSV for analysis.

**Key observations:**
- Semantic similarity is the strongest signal; ROUGE is lower because the LLM paraphrases rather than copying text verbatim (which is correct behavior)

- Questions about specific numerical facts (TOPS figures, release years) score highest on composite; broad conceptual questions score lower due to ROUGE penalizing valid paraphrasing

## 🚀 Quick Start

### 1. Clone & Install

```bash
git clone https://github.com/YOUR_USERNAME/rag-npu-domain.git
cd rag-npu-domain

python -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate

pip install -r requirements.txt
```

### 2. Configure API Key

```bash
cp .env.example .env
# Edit .env and add your Groq API key:
# GROQ_API_KEY=your_key_here
```
Get a free Groq API key at: https://console.groq.com

### 3. Ingest Documents

```bash
python main.py ingest
```
This builds the FAISS vector store in `vector_store/` (~5 seconds).

### 4. Run Queries

```bash
# Single query
python main.py query "What is a systolic array?"

# Interactive mode
python main.py interactive
```

### 5. Run Evaluation

```bash
# Automated evaluation (all 15 questions)
python main.py evaluate

# With human rubric scoring
python main.py evaluate --qualitative
```

### 6. Jupyter Demo

```bash
jupyter lab demo.ipynb
```

## ⚠️ Challenges & Lessons Learned

### 1. Chunk Size vs. Retrieval Precision
**Challenge:** Large chunks improved answer quality but hurt retrieval precision — a chunk about "architecture" might contain enough text on other topics to rank for unrelated queries.  
**Solution:** 400-character chunks with 80-character overlap (20%) struck the best balance. Smaller chunks (200 chars) produced retrieval scores 12% higher but answer quality suffered from missing context.

### 2. ROUGE Penalizes Correct Paraphrasing
**Challenge:** The LLM correctly answers questions but in its own words, producing low ROUGE scores even for factually correct responses. ROUGE was designed for summarization where the expected output is a subset of the input.  
**Lesson:** Semantic similarity is a more faithful metric for RAG evaluation than ROUGE alone. The composite score mitigates this by weighting semantic similarity highest.

### 3. Groq Rate Limits During Evaluation
**Challenge:** Running 15 consecutive queries hit Groq's rate limits during evaluation.  
**Solution:** Added a small delay between evaluation queries and batching approach in the evaluator.

### 4. Out-of-Domain Robustness
**Challenge:** Early versions of the system hallucinated plausible-sounding NPU "facts" when asked questions the documents didn't cover.  
**Solution:** The system prompt explicitly instructs the model to state when context is insufficient, tested with deliberate out-of-domain queries (e.g., "What is the recipe for chocolate cake?").

### 5. Source Attribution Accuracy
**Challenge:** A question answered by information spread across multiple documents would miss the `source_hit` metric if we required all sources to be retrieved.  
**Solution:** Changed `source_hit` to check for at least one correct source document in the top-k results, which is both fairer and more practically meaningful.
