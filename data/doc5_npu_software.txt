Title: NPU Programming Models and Software Stacks

The gap between raw NPU hardware capability and what developers can practically achieve depends almost entirely on the software stack. Programming NPUs directly is notoriously difficult: developers must manage scratchpad memory explicitly, schedule data movement to hide latency, and express computation in terms of the hardware's native operations. As a result, nearly all NPU programming happens through high-level ML frameworks and compilers rather than bare-metal code.

The dominant compilation strategy for NPUs is graph-level optimization. ML frameworks like TensorFlow and PyTorch represent neural networks as computational graphs of operations. Compilers such as XLA (Accelerated Linear Algebra, developed by Google), TVM (Tensor Virtual Machine, from the Apache open-source project), and MLIR (Multi-Level Intermediate Representation, a compiler infrastructure from Google) transform these graphs through a sequence of optimization passes: operator fusion, layout transformation, tiling, and quantization. The optimized graph is then lowered to hardware-specific code generation.

XLA was the first major production graph compiler for NPUs, developed initially for Google's TPU. XLA fuses consecutive operations to eliminate intermediate memory allocations, tiles large matrix multiplications to fit within on-chip SRAM, and applies algebraic simplifications. The XLA just-in-time (JIT) compiler traces PyTorch or JAX programs and compiles entire computation graphs to TPU binaries. XLA's ahead-of-time (AOT) compilation mode is used for production deployment where compilation latency is unacceptable.

Apache TVM is the leading open-source NPU compiler, supporting targets ranging from Arm Cortex-M microcontrollers to NVIDIA GPUs to custom ASIC accelerators through its VTA (Versatile Tensor Accelerator) backend. TVM introduces the concept of a schedule, which separates the algorithm (what to compute) from the schedule (how to compute it: tiling factors, vectorization, loop order). AutoTVM and Ansor are TVM's automated schedule search systems that use machine learning to discover high-performance schedules without manual tuning.

MLIR represents the current frontier of compiler infrastructure for heterogeneous AI hardware. Unlike previous compilers tied to specific frameworks or targets, MLIR provides a flexible intermediate representation with multiple levels of abstraction — from high-level tensor operations down to hardware-specific dialects. Vendors including Intel, AMD, Qualcomm, and Samsung have developed custom MLIR dialects for their NPU architectures, enabling a common compiler infrastructure while maintaining hardware-specific optimizations.

At the framework level, ONNX (Open Neural Network Exchange) has become the lingua franca for model portability. Models trained in PyTorch can be exported to ONNX format and then compiled by any ONNX-compatible runtime: Microsoft's ONNX Runtime, Qualcomm's SNPE, or Intel's OpenVINO. ONNX Runtime includes an execution provider (EP) architecture that dispatches operations to available accelerators, including NPUs, with a fallback to CPU for unsupported operations.

The emergence of large language models (LLMs) has created new challenges for NPU software stacks. LLM inference has a distinct computational profile: the prefill phase (processing the input prompt) is compute-bound and maps well to systolic arrays, while the decode phase (generating tokens one at a time) is memory-bandwidth-bound and exposes the limitations of many NPU memory subsystems. Techniques like continuous batching, KV cache quantization, and speculative decoding are being integrated into NPU software stacks to address these challenges and make LLM inference economically viable on edge hardware.
